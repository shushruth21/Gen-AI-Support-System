✅ Workflow

Multi-Stage Approval System

📥
1. Ticket Submitted
Customer or internal team member creates a work item. AI triage fires immediately within 15 seconds.
🤖
2. AI Assessment
Gemini evaluates priority, severity, and root cause. If risk is HIGH, the system auto-flags for approval workflow.
👨‍💼
3. Approver Notified
Designated Security Approver receives the request with full context: AI summary, confidence, impacted component.
✅
4. Decision Recorded
Approver accepts, rejects, or reassigns. Full decision + notes stored in approval_events with timestamp.
🚀
5. Execution + Audit
Approved actions trigger remediation. Every step is immutably logged in activity_log for compliance audit.

💡 Learnings

Challenges & Key Takeaways

⚔️ Challenges Faced
🗄️
Schema Evolution
Multi-table migrations with cascading FKs required careful ordering of 11 sequential migration files to avoid constraint violations.
🔗
Frontend ↔ Backend Type Sync
Pydantic schema mismatches caused runtime 422 errors. Solved by strict TypeScript generics aligned to Pydantic response models.
🔐
RLS Policy Conflicts
Row Level Security blocked seed scripts and API calls. Required service_role key for seeding and anon key handoff for client queries.
🧪
Realistic Data Generation
Building a deterministic-yet-realistic seed required MD5-based pseudo-random seeding to ensure reproducibility across runs.
🏆 Key Learnings
🤖
AI as a Service Layer
Treating AI enrichment as a service-layer concern (not DB-layer) enabled graceful degradation when model confidence was low.
📊
JSONB for Flexible AI Metadata
Storing AI output as JSONB allowed evolving the schema without migrations — critical for rapidly changing LLM output formats.
🏗️
Server Components for Data Freshness
Next.js 14 Server Components simplified live data fetching — no client-side state management needed for read-heavy dashboards.
🔄
Audit-First Design
Building activity_log and ai_audit_log from day one proved invaluable for debugging AI decisions and demonstrating system transparency.
